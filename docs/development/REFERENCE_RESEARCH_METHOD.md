# 外部参考学习与方案生成法（Reference-Driven Engineering）

本文档定义：当 OpenCove 面对一个在行业内大概率已经被别人做过、讨论过、踩过坑的问题类时，应该如何主动查找外部参考、提炼经验、结合本仓约束形成方案，并把结果沉淀为可复用资产。

默认心智不是“用户要求了才去看别人怎么做”，而是：**只要问题不是 OpenCove 独有，就先判断外部是否已有成熟实践，再带着本地约束做方案。**

目标不是“抄一段别人代码”，而是把外部项目的**承诺、状态建模、边界所有权、trade-off 与回归策略**转化为 OpenCove 自己的一致实现。

## 1. 什么情况下要默认先看外部参考

满足任一条件时，都应默认先做参考学习，再进入本地方案设计：

- 问题大概率属于行业常见问题类，而不是本项目独创难题。
- 主流产品、框架、库、协议或平台通常已经有既定做法。
- 需求涉及用户习惯、交互心智、编辑器 / 画布 / 表单 / 导航行为。
- 需求涉及 auto、heuristic、recovery、retry、sync、cache、auth、session、search、tree、table、import/export、CLI DX 等常见模式。
- 本地 patch 已开始出现“修一个 case、坏一片相邻 case”的迹象。
- 方案选择明显依赖 trade-off，而不是单纯的本仓私有业务规则。

一句话判断：**如果这是一个“别人多半已经踩过坑”的问题，就不要假设自己是第一个遇到它的人。**

## 2. 核心原则

### 2.1 把外部参考当成默认输入，不当成附加项

对成熟问题类，外部参考不是“有空再看”的锦上添花，而是方案质量的一部分。

一个合格方案，至少应该说明：

- 行业内常见做法是什么；
- 哪些做法可以迁移到 OpenCove；
- OpenCove 的约束是什么；
- 为什么最终选择当前方案，而不是另一个看似也能工作的做法。

### 2.2 学“承诺”和“约束”，不要抄“表面实现”

最值得借鉴的通常不是某个阈值、某个函数名、某个类结构，而是：

- 它向用户 / 调用方 / 运维者承诺了什么稳定行为；
- 它如何划分 `state owner` 和 `authority owner`；
- 它在哪些地方故意保守，而不是自作聪明；
- 它如何给出 fallback、manual override 和回归保障。

### 2.3 外部经验必须先转译，再落实现

外部方案到 OpenCove 之间，至少隔着这些差异：

- Electron / 浏览器 / OS 行为差异；
- React / 状态管理 / 渲染时序差异；
- 本仓 context、owner、persistence、IPC、测试基线差异；
- 现有抽象与依赖边界差异。

因此，外部参考只能提供**问题空间、设计约束、常见 trade-off 与回归思路**，不能直接替代本仓实现。

## 3. 参考源优先级

按以下顺序查找，避免直接沉迷二手博客和随手代码片段：

1. **产品 / 帮助中心 / 官方规范 / 官方行为文档**
   - 作用：确认外部项目到底向用户或调用方承诺了什么。
   - 适合回答：默认行为、固定模式、手动 override、兼容语义。
2. **官方示例 / 官方源码 / 官方测试 / RFC / 设计文档**
   - 作用：确认承诺如何映射到状态、边界、默认策略和测试。
   - 适合回答：谁是 owner、默认值是什么、哪些路径是高优先级。
3. **issue / discussion / changelog / maintainer comment**
   - 作用：确认历史坑、平台差异、为什么没有选择另一个更“聪明”的做法。
   - 适合回答：trade-off、限制、误判来源、兼容性考虑。
4. **OpenCove 现有抽象与回归测试**
   - 作用：确认最终应该复用哪里、落在哪一层、避免平行再造。
   - 适合回答：用哪个 hook / service / contract / context，而不是新开一条解释路径。

## 4. 标准执行流程

### Step 1：先把问题写成“问题类”

先用两句话描述：

- `这属于哪一类已有行业先例的问题？`
- `OpenCove 当前真正不稳定 / 不清晰 / 不可预测的承诺是什么？`

示例：

- 交互类：鼠标滚轮有时缩放、有时平移，说明输入语义不稳定。
- 状态类：恢复流程有时把短暂的 runtime observation 当成 durable truth。
- 接口类：分页 / 搜索 / 筛选组合后，调用方无法预测参数优先级。

### Step 2：收集 2-5 个高质量参考，不拼数量

优先覆盖不同层次，而不是同层重复：

- 一个定义外部承诺的参考；
- 一个解释默认实现或测试的参考；
- 一个解释 trade-off 的讨论 / changelog；
- 必要时，再补本仓已有抽象。

记录时重点关注：

- 默认行为是什么；
- 哪些情况允许自动判断，哪些情况必须保守；
- 谁拥有最终写权 / 切换权 / 决策权；
- 当系统不确定时，是切换、保持现状，还是交给用户显式决定。

### Step 3：每个参考只提炼五类信息

不要整页搬运，最多提炼：

1. `承诺（Promise）`
2. `state / authority owner`
3. `1-3 invariants`
4. `fallback / override`
5. `trade-off / 不直接照搬的部分`

推荐模板：

```md
参考：<项目 / 页面 / 文件>
- 承诺：
- state owner：
- authority owner：
- invariants：
  -
  -
- fallback / override：
- 可迁移原则：
- OpenCove 中不能直接照搬的部分：
```

### Step 4：把外部经验转译为 OpenCove 方案

提出方案时，必须显式写出这条链路：

`行业共识 -> 可迁移原则 -> OpenCove 约束 -> 本地设计`

常见转译模式包括：

- **交互 / 事件类**：`raw signal -> normalized observation -> policy/classification -> semantic execution -> UI projection`
- **状态 / 生命周期类**：`external event -> normalized fact -> policy decision -> state transition -> side-effect orchestration -> UI projection`
- **接口 / 数据类**：`input contract -> normalization -> domain rule -> persistence / transport mapping -> caller-visible projection`

目的都是同一个：**不要让原始输入、瞬时观测或局部实现细节直接驱动最终语义。**

### Step 5：自动化 / 启发式默认保守

凡是涉及 `auto-detect / auto-switch / auto-retry / auto-recover / heuristic` 的地方，默认采用保守策略：

- **高置信才切换**：只有明确证据才改变模式、状态或语义。
- **歧义保持稳定**：模糊样本默认保持当前稳定行为，而不是随时翻转。
- **保留显式兜底**：在可能时提供手动 override、固定模式、明确配置或清晰恢复路径。

优秀项目通常追求的是“稳定、可解释、可回退”，不是“自动判断尽可能聪明”。

### Step 6：提出方案时必须给出对比结论

最终方案至少回答这四件事：

- 行业内常见做法是什么；
- 我们这次采用哪些原则；
- OpenCove 与外部参考最大的差异约束是什么；
- 为什么当前方案比其他备选更合适。

如果这四件事说不清，说明参考学习还没有真正转化为设计。

### Step 7：验证按最低有效层落地

验证目标不是“把所有层都跑一遍”，而是用最低 meaningful layer 证明关键承诺成立：

- `Unit`：规则、invariant、normalization、边界条件。
- `Contract`：参数、协议、边界校验。
- `Integration`：owner、持久化、生命周期、跨层协作。
- `E2E`：关键用户路径与真实交互链路。

原则：**先测你从外部参考里学到的那条规则，再测 OpenCove 本地落地后的关键路径。**

### Step 8：最后资产化

至少沉淀为以下之一，最好同时具备：

- 一条专项文档规则；
- 一组回归测试；
- 一段 owner / invariant 说明；
- 一条 review checklist 或设计约束；
- 一份对比型方案摘要。

如果结论只存在于聊天记录、脑内记忆或 commit message，视为没有完成学习闭环。

## 5. 推荐输出模板

当一个任务命中了“先参考行业做法”这一心智，最终输出建议包含：

### 5.1 外部共识

- 行业内常见做法是什么；
- 哪些做法其实是分歧点；
- 哪些做法已经形成成熟心智或事实标准。

### 5.2 本地约束

- OpenCove 的 `state owner / authority owner` 是谁；
- 现有抽象、边界和依赖限制是什么；
- 哪些外部做法不能直接照搬。

### 5.3 本地方案

- 最终采用的原则；
- 关键不变量；
- fallback / override；
- 主要 trade-off。

### 5.4 验证方案

- 哪些规则在 `Unit` 层验证；
- 哪些关键路径在 `Contract / Integration / E2E` 验证；
- 还有哪些限制或补齐项。

## 6. 常见反模式

以下做法都应主动避免：

- **只有用户点名才去参考外部实现**：把参考学习当成被动响应，而不是默认能力。
- **只看代码不看承诺**：最后得到一个局部逻辑成立、整体体验不稳定的实现。
- **直接抄阈值 / 抄结构 / 抄 API 形状**：忽略平台差异和本仓边界。
- **没有转译步骤**：外部代码直接进入本仓，导致 owner、语义、边界都错位。
- **没有比较备选方案**：最终方案像“拍脑袋”，而不是基于共识与约束的取舍。
- **把自动化做得过于激进**：系统比用户更有主见，最终行为不可预测。
- **只修一个 case，不修 bug class**：没有把外部经验沉淀成更稳固的规则。
- **只留聊天总结，不落测试和文档**：结论无法复用，也无法解释为什么这么做。

## 7. 画布输入自动切换：一个具体案例

这是一个具体示例，不是该方法的唯一适用范围。

### 问题类

- 行业先例：编辑器 / 画布 / 设计工具普遍都要处理鼠标、触控板、缩放、平移、框选之间的关系。
- OpenCove 问题：滚轮输入会在缩放与平移之间抖动，说明单次模糊样本被错误地当成了模式切换依据。

### 外部共识

- `auto` 不是无限聪明，而是保守的；
- 高置信信号比单次模糊样本更重要；
- 默认行为必须稳定、可解释；
- 用户应能手动固定为某种输入习惯。

### 转译为 OpenCove

- `mode owner`：只有高置信输入信号能切换 `mouse / trackpad / auto` 语义。
- `semantic exclusivity`：同一连续输入只能落入 `pan` 或 `zoom` 等一种主语义。
- `auto` 策略：模糊普通 wheel 保持当前模式；高置信样本才切换；允许手动固定。
- `selection owner`：`Shift + click`、`Shift + drag marquee` 的反选统一由 selection 语义处理。

### 验证

- `Unit`：验证大幅单轴 wheel 仍为鼠标、轻微噪声不误判、高置信样本可立即回切。
- `E2E`：验证先进入 trackpad 语义，再来鼠标 wheel，画布立即恢复缩放；验证 `Shift` 点击 / 框选反选语义。

## 8. 复盘时该问的几个问题

交付前，用下面问题自查：

- 这是一个行业已有先例的问题吗？如果是，我有没有主动看外部参考？
- 我学到的是对方的“承诺和约束”，还是只抄了一个实现细节？
- 我是否已经说清了 `state owner`、`authority owner` 和关键 invariants？
- 方案里是否明确写出了 `行业共识 -> OpenCove 约束 -> 本地设计`？
- 自动化部分是否足够保守，并提供了显式兜底？
- 如果同类问题再来一次，现在的文档和测试是否能挡住？

## 9. 一句话原则

**只要问题类在行业里已有成熟先例，就先学习别人如何承诺、建模、取舍和回归，再结合 OpenCove 约束提出自己的方案。**
